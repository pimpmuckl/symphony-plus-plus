# Findings: SYMPP-P4-001

## Discovery

- `AGENTS.md` requires one package per PR, tight scope, concrete PR template evidence, no live Linear state, and no raw secrets in files/logs/PR/review text.
- Package spec requires a one-command/request standalone WorkPackage creation path with inputs for kind, repo, base branch, title, product description, engineering scope, acceptance criteria, and review-suite template.
- P1-002 is present as `AccessGrants.Service.mint_worker_grant/3` and stores only `secret_hash`; the raw `WorkKey.secret` is available only in the mint return.
- P1-003 is present as `Policies.Templates.expand/1`; current templates include `quick_fix`, `hotfix`, `standard_pr`, `phase_child`, `investigation`, plus P3 worker kinds.
- P1-004 is present as `Planning.Renderer.render_all/2` and virtual files include context, task plan, findings, progress, acceptance, review suite, and handoff.
- Existing operator-facing Symphony++ command pattern is `Mix.Tasks.Sympp.Mcp`; no product-specific HTTP API exists yet except observability routes.

## Decisions

- Implement service API plus Mix task instead of HTTP: the package allows command or endpoint, and a Mix task fits the existing Symphony++ command pattern without widening web/API scope.
- Treat `policy_template` / `review_suite_template` input as an explicit consistency check against the selected kind for now. The existing schema has no separate policy field, and P4-002 owns broader policy-template expansion.
- Create initial pending plan nodes from the requested engineering scope and acceptance/review expectation so `task_plan.md` is not empty on first render.

## Risks

- Real secret-dependent validation will be covered with generated test secrets only. No real worker secrets will be printed in planning, PR, or final text.
- `mix credo --strict` currently reports three pre-existing findings in `lib/symphony_elixir/symphony_plus_plus/mcp/server.ex` outside this package's touched files. The touched-file Credo lane passes with no issues.

## Review Findings

- T1 Alpha found two valid request-validation issues. Invariant: explicit template fields should match the selected policy, and acceptance criteria should be nonblank if provided. Owner/source of truth: `CreateWork.parse_request/1` backed by `Policies.Templates.expand/1`. Sibling paths checked: Mix task parsing delegates to `CreateWork.parse_file/1`, and worker MCP rendering consumes the created ledger state. Structural fix: accept either the kind or resolved template name for template fields, and reject blank/non-string criteria instead of dropping them. Regression coverage: `create_work_test.exs` covers `mcp` with `policy_template: worker_package` and blank criteria rejection.
- Follow-up review found a valid mixed-alias edge case where `policy_template` and `review_suite_template` could each use different valid aliases for the same resolved policy. Structural fix: validate each provided template field independently against `{kind, resolved_template}`. Regression coverage: the `mcp` request test now supplies both `review_suite_template: mcp` and `policy_template: worker_package`.
- T2 found four valid final-readiness issues: standalone packages were returned in non-dispatchable `created` status, `phase_child` could be created without a parent, the Mix task expanded SQLite special database names, and the task did not explicitly start Ecto SQL before starting the repo. Structural fix: create standalone packages directly as `ready_for_worker`, reject `phase_child`, preserve `:memory:`/`file:` SQLite names while preparing file URI parent dirs, and call `Application.ensure_all_started(:ecto_sql)` before repo startup. Regression coverage: service tests now assert ready status plus phase-child rejection, and Mix task tests cover SQLite special database preservation.
- Cached T2 rerun Bravo found two valid workflow issues: omitted `--database` used a cwd-derived ledger, and `package_acceptance` kinds could be created with no acceptance criteria. Structural fix: the Mix task now seeds the default workflow path from the Mix project `WORKFLOW.md` when no workflow path is already configured, and create-work rejects empty criteria for policies that require `package_acceptance`. Regression coverage: Mix task default database resolution and `mcp` missing-criteria rejection.
- Cached T2 rerun Bravo found two valid edge cases: an already-configured workflow path could still redirect omitted-`--database` creates to another ledger, and blank explicit `id` values produced unusable resource URIs. Structural fix: omitted-`--database` now always selects the Mix project `WORKFLOW.md`, and create-work trims explicit IDs while rejecting blank IDs. Regression coverage: prior workflow override and explicit-ID normalization/rejection.
- Cached T2 rerun found two valid contract/runtime issues: explicit template fields were accepted but did not select the effective policy, and default database resolution temporarily mutated global workflow state without restoring it. Structural fix: `policy_template`/`review_suite_template` now select the effective policy and must resolve to the same policy, while omitted-`--database` resolution restores the prior workflow path after computing the ledger path. Regression coverage: policy selection/conflict tests and workflow-path restoration in the Mix task database default test.
- Cached T2 rerun found a valid downstream consistency issue with cross-kind policy overrides: persisted `kind` drives later policy lookup, so a standalone request cannot safely select a different policy template than its kind. Structural fix: `policy_template`/`review_suite_template` are now consistency assertions for the selected `kind` and its resolved template, not cross-kind overrides. Regression coverage: `kind: mcp` accepts `worker_package`/`mcp`, while implicit `quick_fix` plus `worker_package` is rejected.
- Cached T2 rerun found two valid initial-planning content issues: investigation packages got acceptance-criteria instructions despite not requiring acceptance criteria, and whitespace-only engineering scope suppressed the fallback guidance. Structural fix: acceptance instructions are seeded only for policies with `package_acceptance`, and scope plan body treats blank strings as missing. Regression coverage: investigation creation with blank scope.
- Cached T2 rerun found valid CLI/input edge cases: omitted `--database` still honored a global `:sympp_repo_database` override, blank explicit `--database` could become an unintended filesystem path, and YAML `acceptance_criteria:` decoded as `nil` was rejected for optional-policy requests. Structural fix: default database resolution temporarily ignores/restores `:sympp_repo_database`, blank explicit database args fail usage validation, and nil acceptance criteria normalize to an empty list.
- Cached full-diff T2 rerun found one valid scope-preservation issue: documented hotfix requests could include `allowed_file_globs`, but create-work accepted and dropped that field. Structural fix: WorkPackages now persist `allowed_file_globs` as a string-list column, create-work normalizes and returns it, and `context.md` renders the requested constraints.
- Cached full-diff T2 rerun on the `allowed_file_globs` fix found two valid issues: invalid CLI request files could still start/migrate a ledger, and glob values were rendered as active Markdown. Structural fix: the Mix task now parses the request before repo startup/migration, and `allowed_file_globs` render through the same inert source-text helpers as other operator-supplied fields.
- Cached full-diff T2 rerun after planning closure flagged acceptance-less quick-fix persistence. Existing investigation coverage showed empty criteria persisted, so the code path was sound; added explicit quick-fix omitted-criteria regression coverage to lock the contract.
- Cached follow-up for the acceptance-less quick-fix regression found the first test only asserted the creation response. Structural test fix: reload the WorkPackage from the ledger and render `acceptance.md` from repository state before asserting the no-criteria contract.
- Cached full-diff T2 rerun found malformed explicit `kind` values defaulted to `quick_fix`, and explicit template selection was only useful when a matching kind was also present. Structural fix: absent `kind` may now default from a known explicit policy/review template, while explicit blank/non-string `kind` is rejected and explicit cross-kind templates remain rejected for persisted-policy consistency.
